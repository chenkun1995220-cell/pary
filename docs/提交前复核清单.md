# 提交前复核清单（面向周报与自动化稳定性）

目标是让每次代码/数据调整后，能稳定输出周度结论，不把不确定判断当成正式结果。

## 0) 固定前提（每次执行都确认）
- 设定此次复核对应的运行版本（`git diff` 只包含本次会改的文件）。
- 明确本次改动类型：  
  - 核心代码改动  
  - 自动化脚本改动  
  - 文档与流程改动  
  - 数据补充或样本更新

## 1) 交易闭环脚本健康（核心检查）
- 依次执行：
```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\run_self_analysis.ps1
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\show_automation_check.ps1
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\show_weekly_action_items.ps1
```
- 确认 `outputs/automation/latest_self_analysis.json`、`latest_automation_check.json`、`latest_weekly_action_items.json` 可读且 JSON 结构正常（至少有 `as_of_date`、`automation_status`/`status`、`automation_priority_actions`、`automation_check_schema`）。

## 2) 周期性运维状态（每周前先复核）
- 执行：
```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\run_weekly_ops_check.ps1
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\show_weekly_ops_history.ps1
```
- 核对 `outputs/automation/latest_weekly_ops_check.json` 的字段：
  - `weekly_ops_check_status` 处于 `ready / needs_attention` 可接受  
  - 不允许长期出现 `needs_fix` 且无解释  
  - `external_input_blocker_count` 与 `external_input_blockers` 必须存在，且与 `latest_automation_check.json` 中的外部输入阻塞保持一致
- 将历史摘要中的 `recurring_health_issues` 与建议动作与当前本周现象对齐

## 3) 周结论与交付验收（输出有效性）
- 执行：
```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\show_weekly_conclusion.ps1
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\run_weekly_delivery_check.ps1
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\show_weekly_delivery_history.ps1
```
- 核对 `outputs/automation/latest_weekly_conclusion.json` 与 `latest_weekly_conclusion.md` 是否同步更新且字段齐全：  
  - `valuation_targets`  
  - `conclusion_payload`  
  - `candidate_summary`  
  - `overall_health`  
- 核对 `outputs/automation/latest_weekly_delivery_check.json` 的关键状态：
  - `delivery_check_status` 不应是 `critical`
  - `conclusion_signal_status` 不能长期是 `needs_fix`
  - 若出现 `missing_conclusion_signals`，先补充说明并按建议重跑 `show_weekly_conclusion.ps1`

## 4) 数据路径与时间新鲜度（防止“旧输出复用”）
- 核对三市场 `latest_run_summary.md` 时间线是否一致、最新周报日期是否在允许窗口（默认 8 天以内）。
- 核对主要输出文件都存在：
  - `outputs/us_universe/latest_run_summary.md`  
  - `outputs/cn_universe/latest_run_summary.md`  
  - `outputs/hk_universe/latest_run_summary.md`  
  - `outputs/automation/latest_self_analysis_manifest.json`  
  - `outputs/automation/latest_automation_check.json`  
  - `outputs/automation/latest_weekly_conclusion.md`  
  - `outputs/automation/latest_weekly_delivery_check.json`

## 5) 候选结论与开发治理复核
- 开发流程不绑定具体模型名称或版本；执行前记录运行环境能力、权限、依赖和数据限制。
- 按最小可验证单元完成收敛迭代，并保留日志、输出文件和测试组成的证据三件套。
- 按风险分级复核；影响数据口径、筛选、估值、预测或发布的变更必须具有独立验证证据、完整测试和失败回退策略。
- 确认 `development_execution_profile=capability_adaptive_single_agent`、`review_policy=risk_based_independent_verification`、`model_version_pinned=false`、`upgrade_compatibility_required=true`。
- 模型或环境升级后必须重跑相同质量闸门，不得降低验收标准。
- 进入周更或正式交付前检查：
  - `missing_conclusion_signal_fixes`
  - `automation_priority_actions` 与 `priority_action_details`
  - 人工复核建议与候选优先级是否自洽
- 确认 `latest_candidate_risk_resolution_review.json` 为 `status=ready`、`issues=[]`，且 `manual_pending_count<=manual_pending_limit<=5`。
- 确认 `risk_action_total_count=auto_routed_count+manual_pending_count`，每项都具备估值敏感性、核心风险、买入条件、放弃条件和重新开启条件。
- 目标价触及 60% 保护上限时，必须保留未封顶目标价及低/基准/高三档敏感性；不得把保护上限解释为精确收益预测或自动买入批准。
- 所有模型优化建议必须先进入影子层；只要在自动化信号、交付健康或候选风险描述上出现分歧，优先修正文档 + 运行链路，不得自动修改正式模型参数。
- 复核入口：`docs/中期目标与模型协作规范.md`。

## 6) 决策分级与放行条件（周报发布前）

### 中期目标完成度收尾

- 确认 `outputs\automation\latest_medium_term_goal_review.json` 存在且结构可读。
- 确认 `strategy_code=evidence_prediction_decision_maturity`、`period=8 weeks`。
- 确认 `goals[]` 中每个目标都有 `module` 和 `completion_percent`。
- 确认 `task_closeout_snapshot` 包含 `current_module`、`module_completion_percent`、`medium_term_overall_completion_percent`、`current_target_total_completion_percent`。
- 确认 `environment_compatibility_policy=runtime_capability_detection_with_safe_fallback`，且 `development_governance_note` 明确说明模型无关、能力自适应和升级后重验边界。
- 确认当前 S&P 500 来源产物包含 `missing_ticker_review_queue`，避免只看到缺失计数、看不到需要人工核对的 ticker 明细；当 `recommended_followup=provide_official_constituents_csv` 时，还必须确认 `source_file_request_file` 指向的请求包存在，且 `latest_weekly_action_items.json` 包含 `provide_official_constituents_csv`，便于人工按 dry-run 命令和正式导入命令补源。请求包和每周行动项还必须说明 `source_file_inbox_size_bytes`、`source_file_inbox_sha256`、`source_file_inbox_modified_at` 的当前值，例如 `source_file_inbox_size_bytes: 当前值`、`source_file_inbox_sha256: 当前值或 none`、`source_file_inbox_modified_at: 当前值或 none`，用于确认人工投递的官方 CSV 文件版本。
- 当官方 CSV 是外部输入阻塞时，`latest_weekly_conclusion.json` 必须在 `priority_input_gaps` 中保留匹配的 `provide_official_constituents_csv` 缺口；若遗漏，`run_pre_submit_review.ps1` 会标记 `weekly_conclusion_missing_sp500_external_input_gap`。
- 当 `sp500_current_membership_source_inbox_status.external_input_required=true` 时，确认 `automation_check` 与 `weekly_ops_check` 均同步输出匹配的 `external_input_blockers`；若任一层遗漏，`run_pre_submit_review.ps1` 会标记 `automation_check_missing_sp500_external_input_blocker` 或 `weekly_ops_check_missing_sp500_external_input_blocker`，避免官方 CSV 外部输入阻塞在交付链路中被静默丢失。
- 确认 `task_closeout_snapshot.medium_term_overall_completion_percent` 与 `overall_completion_percent` 一致，且 `task_closeout_snapshot.current_target_total_completion_percent` 与 `current_target_total_completion_percent` 一致；若 `current_module` 能匹配到 `goals[].module`，则 `task_closeout_snapshot.module_completion_percent` 必须与对应目标的 `completion_percent` 一致。
- 若缺少或不一致，`run_pre_submit_review.ps1` 会标记 `medium_term_goal_review_missing_progress_fields`、`medium_term_goal_review_missing_closeout_snapshot`、`medium_term_goal_review_missing_closeout_snapshot_fields`、`medium_term_goal_review_missing_goal_completion`、`medium_term_goal_review_closeout_overall_mismatch`、`medium_term_goal_review_closeout_current_target_mismatch`、`medium_term_goal_review_closeout_module_mismatch`、`medium_term_goal_review_missing_collaboration_boundary` 或 `medium_term_goal_review_collaboration_mode_unsafe`。
- `latest_pre_submit_review.json` 会同步输出 `development_closeout.current_module`、`development_closeout.module_completion_percent`、`development_closeout.medium_term_overall_completion_percent` 和 `development_closeout.current_target_total_completion_percent`，便于每次小任务收尾直接引用。
- 若本次小任务明确属于某个目标模块，可运行 `run_pre_submit_review.ps1 -CloseoutGoalCode <goal_code>`，让 `development_closeout` 输出该目标对应的模块和完成度。

- 允许发布：
  - 本周 `run` 能完整落盘  
  - 周结论报告与交付验收均处于非阻断态  
  - 缺口说明存在但有明确修复动作（不是无上下文“待观察”）

  - 交付健康 `needs_fix`
  - 关键输出缺失（`latest_weekly_action_items` 或 `latest_self_analysis_manifest` 不可读）
