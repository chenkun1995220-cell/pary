## 中期目标

在 8 周内，把项目从“可以自动产出”推进到“可信、可复核、可连续评估”的三市场低估研究系统：`gpt5.3-codex-spark` 代表快速迭代习惯，`gpt5.5` 代表关键复核和正式收口习惯。当前版本还没有自动调度两个模型协作开发，实际执行仍由单 Codex 完成，并通过复核清单模拟 `gpt5.5` 的风险收口角色。  
重点包括：

- S&P 500 历史成分证据验证比例提升到 50% 以上，并持续降低弱证据行数；
- 1 周和 1 个月预测成熟样本达到 30 条以上，再评估方向命中率和超额收益；
- 候选解释能力增强，保持字段完整率并压降风险复核项；
- 三市场数据质量持续收敛，港股数据质量作为重点修复对象；
- 自动化交付链路稳定、异常自动感知、参数与口径变更留痕。
- 将 `gpt5.3-codex-spark` 的快速开发习惯与 `gpt5.5` 的审慎复核习惯组合成固定工程纪律，保证每次迭代都有小步交付、证据留存、正式收口和可回退边界。

中期验收不只看功能数量，还要看开发习惯是否稳定固化：每次改动必须可回放、有证据、可复测、可回退；进入周度发布前必须完成自动化复核，并完成 `gpt5.5` 或同等级复核清单的风险收口。

## 模型协作机制（固定规则）

### `gpt5.3-codex-spark` 口径负责

- 小步快跑，聚焦明确的执行动作：脚本补齐、接口修复、测试补全、文档同步；
- 每次只推进一个可验证子目标，交付后必须留下文件产物和复现证据；
- 优先保证“可运行”和“最小范围正确”，再补齐边界说明。

### `gpt5.5` 口径负责

- 做关键复核与关键口径判断，处理架构一致性、风险边界、跨市场协同问题；
- 对每一次进入周度流程的能力给出 `ready / needs_attention / blocked` 判定；
- 检查报告与指标口径，确认交付可解释性与可追溯性。

### 迭代与收口

- 每次快速迭代结束后，必须经过 `gpt5.5` 口径或同等级复核清单检查后，才能进入周度正式动作；
- 不合格项要求退回实现环节修复，完成后再复核；
- 交付与复核动作不允许互相替代，必须一前一后闭环。
- 所有模型优化建议必须先进入影子层，经过样本外验证、风险复核和人工确认后，才允许形成正式升级建议；
- 自动化、自我分析、回测和周结论不得自动修改正式模型参数。

### 开发习惯（嵌入式治理）

- 每次只交付一个可回放的改动：先改一个任务目标、补一组最小脚本/文档，复测通过后再推进下一个。
- 每个改动都要有“证据三件套”：日志、输出文件、新增测试（有条件时必须包含至少一个回归测试）。
- 避免一次性大改；每次改动前先说明：改了什么、改了哪个市场/模块、预期影响是什么、回退策略是什么。
- 任何影响评分参数、数据口径、告警策略的动作，都必须同步更新 `docs` 与相关自检入口，直到 `run_pre_submit_review` 可复跑。
- 周期性节律固定为：快速迭代口径先跑收敛迭代 → 自动化复测 → `gpt5.5` 口径或同等级复核清单收口判定 → 结果可发布。

### 组合开发习惯（中期目标内置）

- `gpt5.3-codex-spark` 采用“先补最小闭环”的习惯：先让单一功能可运行、可复测、可落盘，再扩展覆盖范围。
- `gpt5.5` 采用“先看正式风险”的习惯：重点复核数据口径、候选解释、回测证据、影子层边界和是否存在误导性结论。
- 两者组合后形成固定顺序：需求确认 → 小步实现 → 本地证据 → 自动化复核 → 关键风险复核 → 周度发布判断。
- 若 `gpt5.3-codex-spark` 产出与 `gpt5.5` 复核结论冲突，以风险更保守的一侧为准，先进入人工复核或影子层，不直接改变正式模型。

当前真实执行模式固定记录为 `single_codex_with_gpt55_review_checklist`。只有未来具备可靠的多模型自动调度、独立上下文传递、审计日志和失败回退能力后，才允许把 `automatic_multi_model_collaboration_enabled` 从 `false` 改为 `true`。

### 组合习惯验收项

- `gpt5.3-codex-spark` 每轮只做一个可回放子任务，必须说明输入、输出、改动文件、验证命令和失败边界。
- 每轮实现完成后先做收敛迭代：修正测试、文档、脚本入口和产物路径中的不一致，不把半成品交给周度流程。
- `gpt5.5` 复核时优先检查正式发布判断：数据是否新鲜、结论是否可解释、风险是否被低估、是否误把影子层建议当成正式参数。
- 每次影响评分、预测、告警、数据口径或自动化任务的改动，必须写清回退策略，并确保回退后不会复用旧输出作为本周结论。
- 只有当提交前复核为 `ready`，且 `gpt5.5` 或同等级复核流程没有发现关键风险时，才允许进入正式周度发布。

## 固定动作（每次改动后）

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\run_pre_submit_review.ps1
```

执行后按规则判断：

- `latest_pre_submit_review.json` 为 `ready` 后可继续；
- `needs_attention` 或 `blocked` 必须先修复再复跑；
- 若出现 `sample_accumulating`，仅允许作为观察记录，不能作为稳定结论直接发布；
- 有未解决风险时，需暂停提交并回到实现环节修复。

## 三市场统一执行纪律

美股 / A 股 / 港股采用同一开发习惯：

1. 先数据质量与口径校验；
2. 再输出候选结论与理由；
3. 最后执行周度收口与提交前复核。

## 中期复盘与优化

- 每周固定进行一次自我分析，输出可优化项；
- 每月复盘评分逻辑、阈值策略、告警策略是否需要调优；
- 优化项优先形成下一周任务，保持稳定节奏，不追求一次性一次到位。
