# 中期目标进度看板

## 当前生效口径

新的中期目标采用方案A：`证据、预测与决策成熟化`，周期固定为 8 周。

看板必须输出：
- `strategy_code=evidence_prediction_decision_maturity`
- `overall_completion_percent`
- `current_target_total_completion_percent`
- 每个目标的 `module` 和 `completion_percent`
- 每个目标的 `target_completion_percent` 和 `completion_gap_percent`
- `task_closeout_snapshot.current_module`
- `task_closeout_snapshot.module_completion_percent`
- `task_closeout_snapshot.medium_term_overall_completion_percent`
- `task_closeout_snapshot.current_target_total_completion_percent`

每次小任务开发完成后，最终汇报必须附上当前开发内容所属模块、该模块完成度和当前目标总完成度。开发治理不绑定具体模型名称或版本；执行主体根据运行环境能力完成任务，并按风险等级执行独立复核、完整测试和失败回退。

本阶段目标完成度为：S&P 500 受限回测证据边界 100%，预测评估成熟化 60%，数据质量收敛 85%，候选公司研究清单成熟化 85%，每周自动交付稳定性 90%，模型无关开发治理 85%。看板通过 `completion_gap_percent` 直接显示各目标距离阶段目标的剩余差距。
候选公司研究清单在候选字段完整、风险覆盖完整、风险已分类且风险行动项全部入队时进入 85% 阶段；新增风险处置产物为 `ready`、自动分流数量与总数一致且待人工深研不超过 5 项时进入 90% 阶段。开发治理在提交前复核 ready、执行配置为 `capability_adaptive_single_agent`、风险复核与升级重验契约完整时，可进入 85% 阶段。

开发收尾时可运行：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File F:\chatgptssd\project2\scripts\show_development_closeout.ps1 -GoalCode model_governance_handoff
```

该入口只读取 `outputs\automation\latest_medium_term_goal_review.json`，用于生成“当前开发内容所属模块 / 该模块完成度 / 当前目标总完成度”的中文摘要。

## 目标

本看板用于把项目未来 6-8 周的中期目标变成每周可检查的进度信号。它不抓取行情、不重新评分、不修改正式模型参数，只读取已经生成的自动化复核产物。

当前中期目标不是“自动调参”或“自动给出强买卖结论”，而是把项目从“可以自动产出”推进到“可信、可复核、可连续评估”。

自 2026-07-11 起，S&P 500 历史证据上限已确认，模块改为“S&P 500 受限回测证据边界”并完成收口。当前成分名单继续使用交叉校验替代来源维持每周筛选，证据等级保持 `secondary`，不得升级为 `verified`；历史缺口继续保留为审计指标，但不再生成每周补证任务。预测成熟样本继续积累，在样本和人工复核达标前不发布正式模型优劣结论。

二期验收重点：

1. S&P 500 当前成分交叉校验替代来源保持可用；历史证据固定为 `limited_verified_only`，不得扩大正式回测样本。
2. 预测评估从 `sample_accumulating` 推进到具备成熟样本的可评估状态。
3. 候选风险动作队列和每周人工待办继续压降，周度结论保持可解释、可复核。
4. 正式模型参数继续保持冻结，所有优化建议先进入影子层和人工复核；只有总完成度达到 75% 以上且预测样本、回测证据同时达标后，才讨论人工升级评审。

## 检查范围

看板固定跟踪 6 个目标：

1. `weekly_delivery_stability`：稳定每周三市场交付。
2. `data_quality_convergence`：数据质量从可监控提升到可解释、可收敛。
3. `candidate_review_convergence`：候选结论变成可排序、可复核的研究清单。
4. `forecast_tracking_maturity`：1 周和 1 个月走势预测进入可评估阶段。
5. `backtest_evidence_quality`：补强回测证据，暂不升级模型。
6. `model_governance_handoff`：固化模型无关、能力自适应、风险驱动复核的开发治理习惯。

## 当前判定逻辑

- 三市场交付链路必须保持 `ready`，否则总体状态为 `blocked`。
- 数据质量目标要求候选阻断数保持 0，可重抓缺口收敛到 0，财务人工复核项降至 40 以下或完成分类。
- 候选结论目标要求字段完整、风险提示覆盖完整，保留原始风险行动总数作为审计事实，并将真正待人工深研项限制在 5 项以内。
- 预测跟踪目标要求最新预测缺失数保持 0，成熟样本达到 30 条以上后再评估表现。
- 回测证据目标要求保留真实缺口和 verified 比例，维持证据上限与受限回测边界，不再追逐已证明不可得的官方历史来源。
- 治理目标要求提交前复核和中期目标协作规范保持可用。

## 模型边界

看板始终输出：

- `formal_model_change_allowed=false`
- `formal_model_upgrade_allowed=false`

只有当预测样本、回测证据、数据质量和候选复核都达到阶段验收条件后，才允许进入正式模型升级的人工评审。

## 财务复核分类口径

数据健康复核现在区分三类数字：

- `manual_financial_review_count`：需要使用财务/估值口径解释的总数量。
- `manual_financial_review_classified_count`：已经有明确分类的数量，例如亏损导致负 PE、负净资产导致 PB 异常。
- `manual_financial_review_unclassified_count`：仍缺少分类、需要继续人工判断的数量。

当 `manual_financial_review_unclassified_count=0` 时，说明这批财务复核项已经完成口径分类；即使总数仍高，也不再把它作为第一优先的未处理缺口。此时中期看板会把数据质量下一步切换到仍未收敛的可重抓缺口或其他真实阻断项。

## 行情重抓缺口口径

数据健康复核现在同时区分原始缺口和仍需行动的缺口：

- `refetch_gap_count`：当前行情/估值字段仍不完整的原始数量。
- `refetch_gap_attempted_count`：已经执行过重抓的数量。
- `refetch_gap_action_required_count`：仍需要继续处理的数量，候选池内缺口或尚未重抓的缺口会进入这里。
- `refetch_gap_unresolved_non_candidate_count`：已经重抓、仍未达到 ready、但不在本周候选池内的残留数量。

当 `refetch_gap_action_required_count=0` 且候选受阻数量为 0 时，说明该类缺口不阻断本周候选交付；看板会把数据质量目标视为按计划推进，并把下一优先动作转向候选风险、预测样本或回测证据。

## 候选风险分类口径

候选解释复核现在把风险复核项拆成两层：

- `risk_review_count`：风险提示需要进入复核视野的总数。
- `risk_classified_count`：已经完成类别识别的数量。
- `risk_unclassified_count`：仍没有识别出风险类别、需要补充规则或人工判断的数量。
- `risk_action_required_count`：即使已分类，仍需要明确人工处理动作的数量，例如无安全边际、预期收益为负或基本面风险。

当前支持的风险类别包括：

- `no_margin_of_safety`：当前缺少安全边际。
- `negative_expected_return`：预期收益为负。
- `wait_for_better_entry`：需要等待更好买点。
- `weak_trend`：走势偏弱。
- `low_valuation_confidence`：估值置信度低。
- `fundamental_risk`：收入、净利润、资产负债等基本面风险。

当 `risk_unclassified_count=0` 时，说明风险项已经完成结构化分类；看板会继续检查 `risk_action_required_count`，把真正需要人工处理的候选单独推进。

## 候选风险动作队列口径

候选解释复核还会把需要人工动作的风险项转成动作队列：

- `risk_action_queue_count`：已经进入动作队列的数量。
- `risk_action_unqueued_count`：需要动作但尚未入队的数量。

当前动作类型包括：

- `defer_research`：无安全边际或预期收益为负，暂缓研究或等待更好价格。
- `manual_fundamental_review`：收入、净利润、资产负债等基本面风险，需要人工复核。
- `classify_manually`：规则暂未覆盖，需要人工分类。

当 `risk_unclassified_count=0` 且 `risk_action_unqueued_count=0` 时，说明候选风险已经完成结构化分类并进入可执行队列；中期看板会把候选解释目标视为按计划推进。

## 候选风险处置口径

`latest_candidate_risk_resolution_review.json` 在动作队列之后进一步区分：

- `risk_action_total_count`：原始风险行动总数，只作审计，不因自动分流而被改写。
- `auto_routed_count`：自动转为继续跟踪或等待安全边际恢复的数量。
- `manual_pending_count`：仍需人工深研的优先项，提交前硬上限为 5。
- `cap_applied_count`：正式目标价触及当前价 1.60 倍保护上限的风险项数量。

每个风险项必须包含估值低/基准/高三档敏感性、核心风险、买入条件、放弃条件和重新开启条件。自动分流不构成买入批准；60% 上限只用于抑制极端估值输出，不代表精确预测未来收益。

## 每周待办压降口径

中期看板会读取 `outputs/automation/latest_weekly_action_items.json`，并在 `weekly_delivery_stability` 目标中展示：

- `weekly_action_items_count`：本周人工处理清单条数。
- `weekly_action_backlog_reduction_plan_status`：当出现 `reduce_weekly_action_backlog` 时，检查是否存在可回溯到该动作的 `backlog_reduction_plan`；可取值为 `ready`、`missing` 或 `not_required`。
- `weekly_action_backlog_reduction_plan_categories`：压降分流计划覆盖的待办分类数量。

这些字段只用于判断每周交付待办是否有可复核的压降安排，不会自动关闭人工事项，也不会修改正式模型参数。

当 `weekly_action_backlog_reduction_plan_status=missing` 时，即使三市场周筛和提交前复核主链路仍为 `ready`，`weekly_delivery_stability` 也会标记为 `needs_work`，并把下一步切换为 `review_weekly_action_backlog_reduction_plan`，要求先补齐待办压降分流证据。

## 行情快照与交付一致性口径

美股行情快照采用 `runtime_output_only` 策略，写入 `outputs/us_universe/market_quotes.csv`，不再作为 Git 跟踪文件。运行摘要必须记录快照行数、行情日期范围和 SHA-256，确保当次输入仍可审计。

周收口在交付验收之后生成 `latest_weekly_artifact_consistency.json`。只有三市场运行摘要均新鲜且属于同一自然日、摘要与候选池数量一致、统一周结论的分市场和总候选数一致、交付验收总数及收口日期一致、行情快照元数据与实际文件一致时，该产物才可为 `ready`。提交前复核将其作为必需输入，因此任何不一致都会阻断开发收口，不会长期留成未解释的整表 Git 改动。

## 连续周日交付验收口径

P0 独立验收自 2026-07-12 起按自然周日累计，要求连续 3 个周日完成三市场交付。验收账本由 `scripts\run_weekly_delivery_streak_review.ps1` 生成：

- `outputs\automation\latest_weekly_delivery_streak_review.json`
- `outputs\automation\latest_weekly_delivery_streak_review.md`
- `outputs\automation\weekly_delivery_streak_history.jsonl`

单个周日只有在三市场运行日期一致、运行开始与结束时间完整、周产物一致性与交付验收均为 `ready`、候选数量匹配、提交前复核通过时才计为成功。港股任务还必须在北京时间 14:15:00 至 14:30:00 启动。相同日期重跑会替换当日记录，不会重复累计；日期中断后连续计数从最新有效周日重新开始。非周日或早于 2026-07-12 的运行只输出 `not_scheduled_day`，不写入历史账本。

看板会展示 `weekly_delivery_streak_status`、`consecutive_sunday_ready_count`、`required_consecutive_sundays`、`successful_sunday_dates` 和 `first_hk_1415_validation_status`。累计不足 3 周时保持 `on_track`，不会提前把 P0 判定为 90%；该验收只确认自动交付稳定性，不修改正式模型参数。

## 回测证据下一步动作口径

`backtest_evidence_quality` 的 `next_action` 会按当前证据闭环状态细化：

- 当 `evidence_ceiling_status=evidence_ceiling_confirmed` 时，下一步固定为 `maintain_limited_backtest`。
- 当当前 S&P 500 成分来源给出 `sp500_current_source_recommended_followup=review_current_membership_source_status` 时，下一步为 `review_current_membership_source_status`，先核对仍缺失的当前成分 ticker 或来源导入覆盖。
- 历史缺口只进入 `membership_evidence_unresolved_gap_count` 审计指标，不进入每周行动队列。

该关闭状态不代表历史成分文件已经补齐或改写，也不允许自动扩样或升级正式模型。
