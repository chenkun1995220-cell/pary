# 自动化身份与配置版本审计设计

## 背景

现有自动化审计会按照固定目录读取四项任务，并校验任务类型、状态、计划时间、执行环境、项目绑定和提示词安全边界，但不会核对配置文件内部的 `id` 与 `version`。配置文件被复制到错误目录、内部 ID 漂移或配置格式升级到未验证版本时，审计仍可能误判为可运行。

## 目标

为全部四项 Codex 自动化任务增加配置头校验：

- 文件内部 `id` 必须与审计预期的任务目录 ID 完全一致；
- `version` 必须为当前已验证且支持的整数 `1`，布尔值 `true` 不视为整数版本；
- 缺失、类型错误或值不匹配均标记为 `needs_attention`；
- 错误信息明确包含 `id` 或 `version`、预期值和实际值；
- 审计结果与 Markdown 报告展示实际配置版本，便于人工确认。

## 兼容策略

任务名称、模型名称、推理强度和具体项目 ID 继续采用已有策略，不因本改动固定当前值。配置格式未来升级为其他版本时，审计应主动阻断并要求先验证审计器与新格式的兼容性，再显式扩展支持版本；不得静默放行未经验证的格式。

## 测试

采用测试驱动方式覆盖：

1. 测试夹具默认生成 `version = 1` 和与目录一致的 `id`；
2. 缺失 ID、错误 ID 和非字符串 ID 均应被阻断；
3. 缺失版本、字符串版本和未支持版本均应被阻断；
4. 有效配置继续通过，报告包含 `version: 1`；
5. 更新周度运维测试中的有效自动化夹具；
6. 运行自动化审计关联测试、真实自动化配置审计和全量测试。

## 非目标与安全边界

- 不修改任何 Codex 自动化配置；
- 不重新运行三市场行情抓取或统一收口；
- 不修改选股、估值或预测模型参数；
- 不改变四项任务的时间、提示词、模型或项目绑定；
- 不降低任何既有质量门槛。
