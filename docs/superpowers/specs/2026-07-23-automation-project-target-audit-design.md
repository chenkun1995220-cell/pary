# 自动化项目绑定审计设计

## 背景

三项市场定时任务已经校验运行时间、工作目录、模型、推理强度和提示词安全边界，但尚未校验 Codex 项目绑定。即使 `cwds` 仍指向本项目，`target.type` 错误或 `target.project_id` 为空也可能导致任务脱离预期项目上下文。

## 目标

在现有 `codex_automation_audit.py` 中为三项 `cron` 市场任务增加项目绑定校验：

- `target` 必须是 TOML 表结构；
- `target.type` 必须为 `project`；
- `target.project_id` 去除首尾空白后必须非空；
- 不绑定具体 `project_id`，允许项目迁移或重新创建后继续使用同一审计规则；
- 心跳验收任务继续使用 `target_thread_id`，不套用项目绑定规则。

## 行为

任一项目绑定条件不满足时，对应市场任务状态改为 `needs_attention`，并在 `issues` 中给出包含 `target.type` 或 `target.project_id` 的明确原因。全部条件满足时，现有审计结果不变。

审计报告在市场任务详情中显示目标类型和项目 ID，便于人工确认，但不把具体 ID 写入验收标准。

## 测试

采用测试驱动方式覆盖：

1. 测试夹具默认生成有效项目绑定，保持现有测试语义不变；
2. 缺失 `target`、错误的 `target.type`、空白 `target.project_id` 均应失败；
3. 不同的非空 `project_id` 均应通过，证明规则没有绑定具体 ID；
4. 运行自动化审计相关测试、周度运维关联测试和全量测试；
5. 对真实 Codex 自动化目录执行审计，预期四项任务继续为 `4/4 ready`。

## 非目标与安全边界

- 不修改任何 Codex 自动化配置；
- 不修改正式选股、估值或预测模型参数；
- 不重新运行三市场数据抓取；
- 不降低现有质量门槛；
- 不把项目 ID、模型名称或推理强度固定为当前值。
