# 影子验证历史幂等化设计

## 背景

`one_week_forecast_shadow_disposition.py` 每次执行都会把当前验证批次追加到
`one_week_forecast_shadow_parameter_validation_history.jsonl`。同一批次重跑不会改变业务内容，
但仍会新增物理行。当前文件有 32 行、8 个逻辑键，其中 24 行是完全相同的重复记录。
逻辑读取会按 `(evaluation_as_of_date, action_code)` 取最后一条，所以正式结论没有被重复计数，
但原始审计文件会持续膨胀，`duplicate_history_key_count` 也会持续上升。

## 目标与边界

- 同一 `(evaluation_as_of_date, action_code)` 且规范化 JSON 内容相同的重跑不得再次追加。
- 同一键但内容发生变化时允许追加，保留纠错记录，并继续由现有逻辑采用最后一条。
- 不修改影子候选分类阈值、正式模型、市场数据、评分结果或交易边界。
- 整理现有历史前必须创建时间戳备份；整理后每个逻辑键保留最后一条。
- 报告中的 `history_records_added` 必须等于本次实际追加的行数。

## 方案比较

1. **相同键和相同内容跳过，内容变化保留修订（采用）**：重跑幂等，同时保留纠错能力。
2. **只按键跳过**：实现更简单，但同批次修订无法进入历史，会丢失有效纠错。
3. **每次运行重写完整历史**：可保持文件紧凑，但每周运行都承担更高写入和中断风险。

## 设计

新增纯函数 `history_rows_to_append(history_rows, new_rows)`：

- 使用现有逻辑键 `(evaluation_as_of_date, action_code)`；无效键不追加。
- 使用排序键、紧凑分隔符的 JSON 字符串作为规范化内容，避免 Python 中 `True == 1` 之类的类型等价误判。
- 先建立每个键的最新规范化内容，再按输入顺序筛选真正新增或发生变化的记录。
- 同一调用内若出现重复新记录，也只保留第一次实际变化。

`build_shadow_disposition` 用筛选后的记录计算 `history_records_added`，并将其与历史组合后执行现有逻辑去重。
CLI 使用同一函数决定 `_append_jsonl` 的输入，避免报告数量和实际写入不一致。

## 历史整理

代码验证通过后，复制当前 JSONL 为带时间戳的 `.bak` 文件。随后调用现有 `logical_history`
得到每个键的最后一条，并原子写回原路径。整理前后分别计算逻辑键和最终记录的规范化摘要，
确认业务逻辑内容不变；备份文件不纳入 Git。

## 测试与验收

- CLI 对相同输入连续运行两次，历史仍只有一行，第二次 `history_records_added=0`。
- 同键内容变化时追加一行，逻辑历史采用修订后的内容。
- 无效键不追加。
- 现有影子处置测试、相关自动化测试及全量测试通过。
- 整理后物理行由 32 降为 8，逻辑键仍为 8，重复行归零，备份存在。
