# 影子历史并发写入保护设计

## 背景

影子验证历史已经支持顺序重跑幂等，但当前 CLI 在“读取历史、判重、追加”之间没有互斥。
两个进程若同时读取同一旧状态，仍可能分别判断为需要追加并写入重复记录；并发写报告还可能造成
`history_records_added` 与最终历史状态不一致。

## 目标与边界

- 同一历史文件的读取、构建、追加、JSON 输出和 Markdown 输出必须在同一排他事务内完成。
- 锁等待必须有上限；超时后非零失败，不得产生部分历史或报告输出。
- 进程正常退出或异常退出后，操作系统自动释放锁。
- 保持顺序重跑幂等、同键修订追加、last-record-wins 和所有分类阈值不变。
- 不引入第三方依赖，不抓取行情、不重新评分、不修改正式模型参数。

## 方案选择

1. **操作系统建议锁（采用）**：Windows 使用 `msvcrt.locking`，POSIX 使用 `fcntl.flock`；自动随文件句柄释放。
2. **`O_EXCL` 临时锁文件**：跨平台简单，但崩溃后需要陈旧锁判断和清理，误删风险更高。
3. **无锁原子重写**：能避免部分写入，但并发修订仍可能互相覆盖，无法保护完整事务。

## 设计

新增 `history_file_lock(history_path, timeout_seconds=30.0, poll_interval=0.05)` 上下文管理器。
锁文件固定为 `<history>.lock`，保留一个字节供 Windows 区域锁使用；锁文件本身可长期存在，锁状态由
操作系统管理。非阻塞尝试失败时按轮询间隔等待，超过截止时间抛出带路径的 `TimeoutError`。

新增 `write_shadow_disposition_files(...)` 事务函数。它先读取不共享写入的计划、验证和性能输入，随后
获取历史锁，并在锁内重新读取历史、构建 payload、筛选待追加行、追加历史、写 JSON 和 Markdown。
CLI 只解析参数并调用该函数，避免锁逻辑与命令行逻辑交织。

## 错误处理

- 锁超时发生在任何输出写入前，CLI 保持非零退出。
- `finally` 中释放系统锁并关闭句柄；不删除锁文件，避免等待者与新建锁文件发生竞态。
- 解锁只在本进程成功获得锁后执行。

## 测试与验收

- 子进程持锁时，第二个锁请求在短超时后抛出 `TimeoutError`；释放后可再次获取。
- 子进程持锁时启动完整写入事务，释放前历史和输出均不存在；释放后事务成功且历史只有一行。
- 相同输入再次运行仍为零新增；同键修订行为不变。
- 聚焦、相关和全量测试通过；九步报告链保持 ready、候选 65、六项版本 1、动作不变。
